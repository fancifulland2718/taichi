
## Taichi 1.7.4 JIT 编译管线深度分析

### 一、JIT 编译全流程概览

Taichi kernel 从 Python 装饰器到最终机器码执行，经历 **7 个关键阶段**：

```
@ti.kernel Python 函数
    │
    ▼
① Python AST 提取与变换 (kernel_impl.py → ast_transformer.py)
    │
    ▼
② KernelCompilationManager::load_or_compile() — 多级缓存查找
    │  命中 → 跳到 ⑦
    ▼
③ CHI IR 编译管线 (compile_to_offloads.cpp) — 10+ 个 pass
    │
    ▼
④ Offload 任务分解 (offload.cpp) — 拆分为独立并行任务
    │
    ▼
⑤ 后端代码生成 (LLVM IR / SPIR-V)
    │
    ▼
⑥ JIT 编译 + 缓存写入
    │
    ▼
⑦ launch_kernel() 执行
```

---

### 二、编译管线各阶段详细分析

#### 阶段 ①：Python 前端 → CHI IR

入口在 kernel_impl.py 的 `Kernel.ensure_compiled()` → `materialize()`：

1. **源码提取**：通过 `inspect.getsource()` 获取被装饰函数的 Python AST
2. **AST 变换**：ast_transformer.py 使用 Visitor 模式，将 Python AST 节点一一映射为 Taichi IR 语句（通过 `ctx.ast_builder` 调用 C++ 侧的 IR Builder）
3. **模板特化**：`TaichiCallableTemplateMapper` 为不同 `ti.template()` 参数值生成不同编译实例

**性能瓶颈**：每个 kernel 首次调用都需要完整的 AST 解析与变换，这是一个**纯串行**的过程。

#### 阶段 ②：多级缓存机制

缓存查找优先级在 kernel_compilation_manager.cpp 中实现：

| 缓存层级 | 位置 | 命中速度 | 实现 |
|---------|------|---------|------|
| **L1 Python 字典** | `Kernel.compiled_kernels[key]` | 最快 (ns 级) | Python dict |
| **L2 进程内内存** | `caching_kernels_` (C++) | 快 (μs 级) | `unordered_map` |
| **L3 磁盘离线缓存** | `~/.cache/taichi/ticache/` | 较慢 (ms 级) | 二进制文件 `{key}.tic` |

- **Kernel Key 生成**：基于 `CompileConfig` + `DeviceCapabilityConfig` + kernel AST 的哈希值
- **离线缓存清理策略**：支持 `never`/`version`/`lru`(默认)/`fifo`，默认上限 100MB
- **元数据文件**：`ticache.tcb`，使用文件锁 `ticache.lock` 保证并发安全

#### 阶段 ③：CHI IR 编译管线（核心耗时阶段）

定义在 compile_to_offloads.cpp，按严格顺序执行：

```
frontend_type_check → lower_ast → compile_taichi_functions
    → eliminate_immutable_local_vars → type_check
    → bit_loop_vectorize (仅 CPU/CUDA/AMDGPU)
    → lower_matrix_ptr → [scalarize]
    → full_simplify (3 次)
    → [auto_diff] (如启用自动微分)
    → check_out_of_bound (如 debug 模式)
    → flag_access → offload
```

然后进入 `offload_to_executable()`：

```
detect_read_only → demote_atomics
    → cache_loop_invariant_global_vars
    → demote_dense_struct_fors → make_cpu_multithreaded_range_for
    → make_thread_local → make_block_local
    → lower_access → demote_operations
    → [scalarize + half2 优化 (CUDA FP16)]
    → type_check (最终校验)
```

**关键观察**：`full_simplify` 被调用了 **至少 3 次**（Simplified I/II/III/IV），每次都包含常量折叠、代数化简等子操作。这是编译时间的主要消耗来源之一。

#### 阶段 ④：后端代码生成差异

| 后端 | 代码生成路径 | 特点 |
|------|------------|------|
| **LLVM 后端** (CPU/CUDA/AMDGPU) | `TaskCodeGenLLVM` → LLVM IR → 机器码 | 需要 LLVM 优化 pass，支持并行编译 |
| **SPIR-V 后端** (Vulkan/Metal/DX12) | `KernelCodegen` → SPIR-V bytecode | 不走 LLVM，`make_thread_local=false` |

核心差异在 SPIR-V 的编译调用（见 spirv/kernel_compiler.cpp）：
```cpp
// SPIR-V 后端直接调 compile_to_executable，且 make_thread_local=false
irpass::compile_to_executable(ir.get(), ..., /*make_thread_local=*/false);
```
而 LLVM 后端只调 `compile_to_offloads`，后续的 `offload_to_executable` 由 codegen 内部控制。

### 三、编译加速机制

#### 1. 并行编译线程池

在 llvm_program.cpp 中：
```cpp
compilation_workers("compile", config_.print_ir ? 1 : config_.num_compile_threads)
```
- 默认 `num_compile_threads = 4`
- 如果开启 `print_ir` 则退化为单线程（避免输出混乱）
- `ParallelExecutor` 实现在 parallel_executor.h，是一个标准的线程池 + 任务队列模型
- **重要**：此并行仅用于 LLVM 后端对多个 OffloadedStmt 任务的并行 codegen
- **SPIR-V 后端不使用**此并行编译机制

#### 2. `TaichiLLVMContext` 线程安全设计

见 llvm_context.h：

```cpp
struct ThreadLocalData {
    std::unique_ptr<llvm::orc::ThreadSafeContext> thread_safe_llvm_context;
    llvm::LLVMContext *llvm_context;
    std::unique_ptr<llvm::Module> runtime_module;
    std::unordered_map<int, std::unique_ptr<llvm::Module>> struct_modules;
};
```
每个编译线程拥有独立的 LLVM Context、runtime module 克隆和 struct module 克隆，避免锁竞争。

#### 3. 可调编译参数

| 参数 | 默认值 | 加速效果 | 副作用 |
|------|--------|---------|-------|
| `opt_level` | 1 | 设为 0 可跳过部分优化 pass | **但 BLS 需要 opt_level≥1** |
| `external_optimization_level` | 3 | 控制 LLVM/SPIR-V 后端优化级别 | 降低会影响运行时性能 |
| `offline_cache` | True | 二次运行秒级启动 | 磁盘空间消耗 |
| `num_compile_threads` | 4 | 增加可加速大 kernel 编译 | 仅对 LLVM 后端有效 |
| `print_ir` | False | 开启会强制单线程编译 | 调试用 |
| `debug` | False | 开启会增加 check_out_of_bound 等 pass | 显著增加编译时间 |
| `fast_math` | True | 允许浮点近似优化 | 可能影响数值精度 |
| `real_matrix_scalarize` | True | 矩阵操作标量化 | 增加 IR 节点数量 |
| `half2_vectorization` | False | CUDA FP16 双元素向量化 | 仅 CUDA 可用 |

### 四、JIT 编译的已知性能问题与可能优化方向

1. **首次编译延迟 (Cold Start)**：即使有离线缓存，首次 kernel 调用仍需完整编译（数百 ms 到数秒级别）。`full_simplify` 的多次调用是主要耗时来源。

2. **SPIR-V 后端无并行编译**：Vulkan/Metal 后端的代码生成是单线程的，对于有大量 offloaded task 的复杂 kernel 尤为明显。

3. **模板特化导致膨胀**：每种不同的 `ti.template()` 参数组合都会触发一次完整编译，如果模板参数空间大，编译开销会线性增长。

4. **AST 变换的 Python 开销**：kernel 的 Python→CHI IR 变换在 Python 层完成（通过 `ast` 模块），比原生 C++ 解析慢一个数量级。

5. **缓存 key 碰撞与失效**：kernel key 基于 AST hash + config，`CompileConfig` 的任何微小变化（如切换 `debug` 模式）都会使整个缓存失效。
