# Phase A.2 — Typed-pointer & legacy PassManager Worklist

Generated by grepping `taichi/` and `tests/cpp/` for LLVM deprecated APIs
that are known to cause build failures on LLVM 16+ (typed pointers) or
LLVM 20+ (legacy PassManager removal).

Scope of Phase A (LLVM 15 → 19.1 LTS): **opaque-pointer cleanup only**.
Legacy PassManager sites are left untouched and the `-Wdeprecated` warning
is suppressed (see `taichi/codegen/llvm/CMakeLists.txt` and
`taichi/runtime/llvm/CMakeLists.txt`). Those sites move to Phase 6 when we
migrate 19 → 22 and switch to the new PassBuilder.

## A.2a — Opaque-pointer migration (required for LLVM 19)

| # | File | Line | Current API | Replacement strategy |
|---|------|------|-------------|----------------------|
| 1 | `taichi/codegen/llvm/codegen_llvm.cpp` | 1803 | `ptr_ty->getPointerElementType()` | Look up pointee via `PtrTypeTracker` (added by this phase) using the SSA value that produced `ptr_ty`. |
| 2 | `taichi/codegen/llvm/codegen_llvm.cpp` | 2334 | `struct_ty->getPointerTo()` | Replace with `llvm::PointerType::get(struct_ty->getContext(), addr_space)` — no element type needed for opaque ptrs. The surrounding `CreateBitCast` can likely be dropped once load/store call sites carry the type explicitly. |
| 3 | `taichi/codegen/llvm/llvm_codegen_utils.cpp` | 32 | `a->getPointerElementType(), b->getPointerElementType()` (x2) | Function is `is_same_type(Type* a, Type* b)`. With opaque pointers all pointers compare equal; this helper degenerates to `a->isPointerTy() && b->isPointerTy()`. Callers must be audited — most rely on it to detect struct layout identity, which must move to a side-channel (PtrTypeTracker or Taichi's own `DataType`). |
| 4 | `tests/cpp/llvm/llvm_offline_cache_test.cpp` | 138 | `llvm_ctx->setOpaquePointers(false)` | Remove the call. The context defaults to opaque; the surrounding test validates bitcode round-trip and must be re-baselined against an opaque-pointer reference file. |

Additional audit steps:

- Any IR-builder call site that previously passed a typed pointer to
  `CreateLoad` / `CreateStore` / `CreateGEP` / `CreateCall` without an
  explicit element type argument will fail to compile on LLVM 19. These
  will surface as compile errors once we flip the LLVM dep. The list
  above is the set of **explicit** deprecated API uses; the **implicit**
  set (overloads that now require an element type) is discovered by the
  first clean build against LLVM 19 and tracked in the PR description.

## A.2b — Legacy PassManager sites (DO NOT touch in Phase A)

These compile fine on LLVM 19 but emit `[[deprecated]]` warnings. Phase A
suppresses the warning via:

```cmake
# in the CMake target that includes these files
if(MSVC)
  target_compile_options(${tgt} PRIVATE /wd4996)
else()
  target_compile_options(${tgt} PRIVATE -Wno-deprecated-declarations)
endif()
```

## Infrastructure added in Phase A.2

`taichi/codegen/llvm/ptr_type_tracker.h` (new, ~40 lines):

- `class PtrTypeTracker` maintains a `DenseMap<Value*, Type*>` that maps
  an SSA pointer value to the element type it was created with.
- Populate in every Taichi-side wrapper around `CreateAlloca`, `CreateGEP`,
  `CreateBitCast`, `CreateCall (returning ptr)`, and on `Argument`
  materialization.
- `Type* get_element_type(Value* ptr)` replaces direct
  `ptr->getType()->getPointerElementType()` calls. Asserts on lookup
  miss so we catch unhandled producers early.

This header is the only new file introduced in Phase A. It has zero
runtime cost and is header-only (the map lives on `TaichiLLVMContext`).
