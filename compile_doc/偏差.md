
## 1. `fast_math` 不对称 — **影响最大**

### 根因

CUDA 后端（通过 LLVM）在两个层面启用了 fast_math：

**层面 A：超越函数选择**（codegen_cuda.cpp）

当 `fast_math=true` 时，CUDA codegen 将 `sin/cos/log` 替换为精度更低的 NVIDIA 快速实现：
- `sin → __nv_fast_sinf`（误差可达 $2^{-21.41}$ ULP，相对精度仅约 6 位有效数字）
- `cos → __nv_fast_cosf`
- `log → __nv_fast_logf`

而 `exp/tan/tanh/asin/acos` 仍使用标准精度 `__nv_expf` 等。

**层面 B：CPU 后端 LLVM 级优化**（[codegen_cpu.cpp](taichi/codegen/cpu/codegen_cpu.cpp#L256-L266)）

CPU 后端的 `optimize_module()` 设置了：
```cpp
options.UnsafeFPMath = 1;    // 允许假设无 NaN/Inf
options.NoInfsFPMath = 1;
options.NoNaNsFPMath = 1;
options.AllowFPOpFusion = FPOpFusion::Fast;  // 允许 FMA 合并
```

**SPIR-V 后端完全没有 fast_math 等价实现**。所有超越函数直接调用 GLSLstd450 标准精度指令，不受 `compile_config.fast_math` 控制。

### 定量影响

| 函数 | CUDA (fast_math=true) | Vulkan (SPIR-V) | 最大相对误差 |
|------|----------------------|------------------|-------------|
| `sin(x)` | `__nv_fast_sinf` ~6位精度 | `GLSLstd450::Sin` ~7位精度 | $\sim 10^{-5}$ |
| `cos(x)` | `__nv_fast_cosf` ~6位精度 | `GLSLstd450::Cos` ~7位精度 | $\sim 10^{-5}$ |
| `log(x)` | `__nv_fast_logf` ~6位精度 | `GLSLstd450::Log` ~7位精度 | $\sim 10^{-5}$ |

在迭代累积场景（如粒子模拟、PDE 求解）中，单步 $10^{-5}$ 误差经过数千步积分后可放大至 **肉眼可见差异**。

### 修正方法

```python
ti.init(arch=ti.cuda, fast_math=False)  # 同时禁用两端的fast_math
```

或者在 Taichi 源码层，在 [codegen_cuda.cpp](taichi/codegen/cuda/codegen_cuda.cpp#L302) 中将 `__nv_fast_sinf` 等替换为 `__nv_sinf`，使 CUDA 在 fast_math 下仍使用标准精度函数。

## 2. f64 超越函数限制 — **影响大**

### 根因

[spirv_codegen.cpp](taichi/codegen/spirv/spirv_codegen.cpp#L949-L960) 中通过 `UNARY_OP_TO_SPIRV` 宏硬编码了最大位宽限制：

```
sin(13, 32)  cos(14, 32)  tan(15, 32)  asin(16, 32)
acos(17, 32)  tanh(21, 32)  exp(27, 32)  log(28, 32)
```

当用户声明 `ti.f64` 后，CUDA 后端能用 `__nv_sin`（f64精度），但 Vulkan 后端直接 **`TI_ERROR` 硬错误**。如果代码以 f32 运行但用户预期某些中间计算保持更高精度，Vulkan 侧只能以 f32 精度完成，而 CUDA 实际上可能通过隐式提升获得更好精度。

仅 `sqrt(31, 64)` / `round(1, 64)` / `floor(8, 64)` / `ceil(9, 64)` 支持 f64。

### 修正方法

- 限定 `default_fp=ti.f32`，避免在 Vulkan 后端使用 f64 超越函数
- 若需要 f64 精度，使用 CUDA 后端或 CPU 后端
- 源码级修复：在 SPIR-V codegen 中为 f64 超越函数实现 Taylor 展开/Polynomial 近似的 polyfill

## 3. 原子操作语义差异 — **影响中等**

### 根因

**浮点原子加法**：

- CUDA：使用 LLVM 原生 `atomicrmw fadd`，SequentiallyConsistent 内存序（[codegen_llvm.cpp](taichi/codegen/llvm/codegen_llvm.cpp#L1531)）
- Vulkan：依赖 `VK_EXT_shader_atomic_float` 扩展（[spirv_codegen.cpp](taichi/codegen/spirv/spirv_codegen.cpp#L1649-L1667)）。若设备**不支持**该扩展，回退到 CAS 循环模拟（[spirv_ir_builder.cpp](taichi/codegen/spirv/spirv_ir_builder.cpp#L1323-L1355)）

CAS 循环实现使用 `OpBitcast` 将 float 转为 uint32 进行比较交换，在高竞争场景下可能因重试次数差异导致 **不同的浮点累加顺序**。

**浮点原子 min/max**：
- CUDA：调用运行时函数 `atomic_min_f32` / `atomic_max_f32`
- Vulkan：无原生支持，全通过 CAS 循环（[spirv_ir_builder.cpp](taichi/codegen/spirv/spirv_ir_builder.cpp#L1342-L1353) 使用 `call_glsl450(FMin/FMax)`）

### 定量影响

浮点加法非结合律 $(a+b)+c \neq a+(b+c)$。当大量线程同时 `atomic_add` 时，累加顺序不同可导致高数量级误差。例如 10 万个 $10^{-3}$ 量级的值求和，不同顺序可产生约 $10^{-4}$ 的差异。

### 修正方法

- 避免大规模浮点 atomic 聚合，改用 hierarchical reduction
- 使用 `ti.f64` 进行累加（但仅在 CUDA/CPU 后端可行）
- 将 `atomic_add` 场景转换为先写入数组再 parallel reduce

## 4. Thread-Local / Block-Local 优化被禁用 — **影响中等**

### 根因

kernel_compiler.cpp 硬编码：
```cpp
irpass::compile_to_executable(ir.get(), compile_config, &kernel_def,
    ..., /*make_thread_local=*/false);
```

而 LLVM 后端通过 `compile_to_offloads` + `offload_to_executable` 路径，会根据 `compile_config.make_thread_local` 和 `compile_config.make_block_local` 启用 TLS/BLS 优化（compile_to_offloads.cpp）。

TLS (Thread-Local Storage) 优化将全局 `atomic_add` 转化为线程局部累加 + 最终一次全局写回。这意味着：
- CUDA：每线程本地累加 n 次，最后 atomic_add 一次 → **更少的浮点舍入**
- Vulkan：每次迭代都全局 atomic_add → **累加顺序完全不同**

这直接改变了浮点运算的结合方式和次数。

### 修正方法

- 在用户代码层手动实现 thread-local 累加模式
- 源码级：在 SPIR-V codegen 中实现 workgroup shared memory 版本的 TLS 优化

---

## 5. Subgroup Reduction vs Warp Reduction — **影响中等**

### 根因

CUDA 后端的 `optimized_reduction()`（codegen_cuda.cpp）调用预编译的 warp-level reduction 运行时函数（如 `reduce_add_f32`、`reduce_min_i32`），这些函数使用 `__shfl_down_sync` 在 warp 内按固定树型规约。

Vulkan 后端的 subgroup reduction（spirv_codegen.cpp）使用 `OpGroupFAdd` / `OpGroupIAdd`，其内部规约顺序**由驱动实现决定**，不保证与 CUDA warp shuffle 的树型规约一致。

### 修正方法

- 这属于浮点非结合性问题，用户层面无法完全消除
- 可以通过降低 `ti.init(default_fp=ti.f64)` 来减小差异的绝对值
- 排序累加值使其按固定顺序规约

---

## 6. 内存布局差异（STD430 vs C 对齐） — **影响小（通常透明）**

### 根因

- SPIR-V 使用 **STD430** 布局：vec3 对齐到 16 字节，struct 最终填充到 16 字节对齐
- LLVM 使用原生 C 对齐：无额外填充

Taichi 内部在数据搬运时进行了正确的布局转换（通过 SNode struct compiler），所以**正常情况不会直接导致数值错误**。但在以下极端场景可能触发问题：
- AOT 模式下手动构造 buffer 数据
- 通过 `ext_arr` 传入的数据未按 STD430 对齐

### 修正方法

- 正常使用 Taichi field 时无需关注
- AOT 场景需根据目标后端选择正确的对齐方式

---

## 7. `compile_config.fit()` 差异化配置 — **影响小**

compile_config.cpp:
```cpp
if (arch_uses_spirv(arch)) {
    demote_dense_struct_fors = true;
}
```

SPIR-V 后端强制启用 `demote_dense_struct_fors`，将 dense struct-for 降级为 range-for。这改变了循环调度方式，可能影响线程分配顺序从而影响浮点累加顺序。

---

## 总结修正建议

| 优先级 | 措施 | 效果 |
|--------|------|------|
| **P0** | `ti.init(fast_math=False)` | 消除超越函数精度差 |
| **P0** | 统一使用 `ti.f32`，不在 Vulkan 上用 f64 超越函数 | 避免硬错误 |
| **P1** | 避免大规模浮点 `atomic_add`，改用 reduction 模式 | 减少累加顺序差异 |
| **P1** | 对精度敏感的计算使用 `ti.f64`（仅 CUDA/CPU） | 减小舍入累积 |
| **P2** | 了解 TLS 优化差异，手动实现局部累加 | 对齐两端行为 |
| **P2** | 源码修复：SPIR-V codegen 增加 fast_math 控制路径 | 根本解决 |

核心结论：**最大的单一差异来源是 `fast_math=True`（默认）对超越函数的影响**。仅关闭 `fast_math` 就能消除绝大多数用户观察到的 CUDA vs Vulkan 数值偏差。

根因 2：atan2(0, 0) 在 Vulkan 上行为未定义 — 高度可能

根因 4：global_tmps_buffer 跨 kernel 调用不清零
runtime.cpp:701-724：全局临时缓冲区只在 初始化时 buffer_fill 一次清零，之后所有 kernel 共享同一块 buffer，无任何重置。

如果您的环绕数计算被 Taichi 编译成多个 offloaded task（如先初始化 → 再规约），且使用了 global_tmps 作中间存储（传递 range-for 的边界、或做全局规约），第二次调用同一 kernel 时，global_tmps 中残留上次的值。

虽然 CUDA 的 LLVM 后端也有类似的共享 global_tmps 设计，但 LLVM 后端的 task 编排方式不同（直接在 host 侧处理 range 边界），可能恰好不触发此问题。

验证方法：在每次 kernel 调用前手动做一次 ti.ndrange 的 trivial 计算，观察结果是否变化。

根因 5：有序浮点比较 + NaN 传播
即使不是 atan2(0,0) 造成的 NaN，其他中间计算如 0.0 / 0.0（两点重合时的方向计算）也可能在 Vulkan 上产生 NaN。

SPIR-V 所有浮点比较都使用有序 (Ordered) 变体（spirv_ir_builder.cpp:1093-1097）：

OpFOrdLessThan(NaN, x) = false, OpFOrdGreaterThan(NaN, x) = false — 任何包含 NaN 的比较都返回 false。

而 CUDA 后端在 fast_math=true 时设置了 NoNaNsFPMath=1（codegen_cpu.cpp），编译器可以假设不存在 NaN，从而使用无序比较，NaN 可能被"忽略"而让比较"碰巧通过"。
